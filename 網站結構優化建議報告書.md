## 0. 摘要結論

目前網站的方向正確，視覺風格已有一致性，資訊架構也能支撐現階段內容。它適合作為嘉義高中管樂隊暨校友管樂團的長期形象網站，而不只是單次音樂會宣傳頁。

但網站現在正位於一個重要分界點：  
若未來只維護少量人物、少量消息、少量相簿，現有純手寫靜態網站足夠；若未來要承接 500 位以上校友、四十多屆校友聯演、上百篇人物與活動資料、長期相簿與歷史考證，現行手寫 HTML 架構會逐漸變得難以維護。

本報告最核心建議是：

> 保留目前純靜態網站與視覺設計，但逐步把內容來源改成資料化、模板化、可檢查、可由 AI 協助新增的靜態內容管理架構。

不建議立即推倒重做，也不建議一開始就導入過重的 CMS 或大型前端框架。較適合的方向是「靜態網站 + 結構化內容檔 + 產生器腳本 + 檢查清單」。

## 1. 目前網站狀況紀錄

### 1.1 網站性質

目前網站是純靜態網站，部署目標為 GitHub Pages，正式網址為 `https://cysh.band/`。  
技術組成為：

- HTML
- CSS
- 原生 JavaScript
- `.js` 資料檔
- WebP 圖片素材
- sitemap、RSS、robots、llms 等靜態 SEO 檔案

目前沒有使用：

- React、Vue、Astro、Eleventy 等建置框架
- 後端伺服器
- 資料庫
- CMS 後台
- npm 建置流程

這讓網站具備三個優點：

1. GitHub Pages 可直接上線，部署成本低。
2. 離線雙擊 HTML 也能預覽，對不熟程式者相對友善。
3. 技術依賴少，未來不容易因套件更新而壞掉。

同時也有三個限制：

1. 大量頁面共用區塊必須手動複製與維護。
2. 內容與版型混在 HTML 裡，AI 或人工新增內容時容易漏改。
3. 資料愈多，愈難追蹤哪裡是主要資料來源。

### 1.4 目前視覺設計狀況

目前視覺風格大致穩定，具有下列特徵：

- 暖白紙質底色。
- 金色作為重點與連結色。
- 深色主要保留在頁尾與少數收尾區塊。
- 標題使用 Noto Serif TC，內文使用 Noto Sans TC。
- 編號、年份、英文字使用 Cormorant Garamond 作裝飾性對比。
- 卡片、時間軸、表格、相簿格線、聯演項目等元件已有一套樣式。

整體氣質適合「歷史型校友社團網站」，不是過度商業化的活動網站。這是目前網站最值得保留的部分之一。

### 1.5 目前內容脈絡狀況

經過先前整理，目前網站已經具備下列內容基礎：

- 名稱與組織：在校生社團、校友團、合稱三者有明確區分。
- 歷史年表：1931、1976、1985、1993、1994、2008、2024、2026 等節點清楚。
- 編號文化：編號結構、聲部代碼、字頭、取得時機等規則清楚。
- 人物網絡：城市推動者、指揮、演奏家、教育者、作曲編曲、社會知名校友等分類已形成。
- 校友聯演：第 1 屆、第 18、21、23、24、26、28-41 屆已有不同程度紀錄。
- 影像館：已能承載相簿、圖說與歷史線索。

內容本身已經有長期資料庫價值，因此網站未來不應只以「多加幾頁」的方式擴張，而應逐步建立內容管理架構。

## 2. 主要風險評估

### 2.1 重複內容風險

同一資訊目前可能散落在多處。例如一位校友可能同時出現在：

- `data/alumni.js`
- `data/number-lookup.js`
- `people.html`
- `people/{編號}.html`
- `concerts.html`
- `concerts/{屆別}.html`
- `news/*.html`
- `gallery/*.html`
- `sitemap.xml`
- `feed.xml`

當資料少時可以手動維護；當資料多時，最容易發生：

- 某人現職在人物頁更新了，名錄卡仍是舊職稱。
- 新增個人頁後，人物誌或名錄沒有連結。
- 聯演頁提到人物，但忘記加個人頁連結。
- 新增消息文章，但忘記更新 `data/news.js`、`feed.xml` 或 `sitemap.xml`。
- 修改導覽或頁尾，部分舊頁忘記同步。

### 2.2 手寫 HTML 擴張風險

目前人物頁、聯演頁、相簿頁多為手寫 HTML。這種方式可控，但不適合大量擴張。

風險包括：

- AI 產生 HTML 時容易漏掉固定 head、nav、footer。
- 不同頁面表格格式可能逐漸不一致。
- SEO meta、OG title、description 容易重複或漏改。
- 手機版樣式若新增特殊 class，容易打破既有響應式規則。
- 長期下來會變成「看得懂頁面，但很難知道資料在哪裡」。

### 2.3 資料公開與隱私風險

`data/number-lookup.js` 有 546 筆資料，若部署到 GitHub Pages，實際上任何人都可以下載檔案。

必須明確確認：

- 這 546 筆是否都同意公開姓名與編號。
- 是否含有不適合公開的在校生或未成年資料。
- 是否含有內部整理用但尚未確認的資訊。
- 是否應分成「公開查詢資料」與「內部維護資料」。

未來如果要 AI 自動化整理名錄，隱私界線必須先被寫進規範與檢查流程。

### 2.4 導覽架構擴張風險

目前 8 項導覽是合理的，但已接近上限：

首頁｜關於樂團｜九十五年｜編號｜人物誌｜校友名錄｜校友聯演｜影像館

未來如果新增：

- 最新消息總覽
- 歷史專題
- 場域地圖
- 文件資料庫
- 聯絡我們
- 捐款或支持
- 管樂節專題

就不適合再把所有項目塞進第一層導覽。應該提前規劃分群。

### 2.5 `concerts.html` 過長風險

目前 `concerts.html` 同時承載：

- 最新消息
- 2026 當屆音樂會
- 歷屆聯演列表
- 場地與演出脈絡

若未來每年都新增屆別，這頁會愈來愈長。建議未來把 `concerts.html` 定位為「總覽」，每一屆的詳細內容都放到獨立頁，總覽只保留摘要與篩選。

### 2.6 `people.html` 手寫卡片風險

人物誌目前是手寫卡片牆，已經有分類與字數規範。未來若人物增加到 50 人以上，手寫卡片會遇到：

- 排序容易錯。
- 分類容易不一致。
- 卡片字數檢查麻煩。
- 有個人頁者的 `<a class="card-head">` 容易漏。
- 名錄與人物誌連結容易不同步。

建議未來改由人物資料自動產生人物誌卡片。

### 2.7 SEO 與 RSS 同步風險

目前新增頁面時需要人工更新：

- `sitemap.xml`
- `feed.xml`
- `data/news.js`
- 可能還有首頁 JSON-LD

這些對讀者不可見，但對搜尋引擎、社群分享與 AI 摘要很重要。人工維護很容易漏，應納入自動化檢查。

## 3. 資訊架構優化建議

### 3.1 短期維持現有 8 項導覽

目前不建議立刻改導覽。現有 8 項足以讓一般訪客理解網站：

- 首頁
- 關於樂團
- 九十五年
- 編號
- 人物誌
- 校友名錄
- 校友聯演
- 影像館

這是清楚且有情感邏輯的結構。短期應保留。

### 3.2 中期改成「主導覽 + 分類頁」

若內容擴張，建議逐步調整為：

| 主導覽 | 下層內容 |
|---|---|
| 關於 | 關於樂團、立案資料、聯絡方式 |
| 歷史 | 九十五年、嘉義管樂之都、場域記憶、歷史專題 |
| 編號 | 編號文化、查號、字頭輪值 |
| 人物 | 人物誌、校友名錄、人物分類 |
| 聯演 | 當屆資訊、歷屆年表、每屆獨立頁 |
| 影像 | 相簿、歷史影像、活動紀錄 |
| 消息 | 最新公告、團練紀錄、相簿更新 |

不一定要立即做下拉選單，但內容概念上要先分清楚。

### 3.3 歷史內容建議分層

目前 `history.html` 是時間軸，適合快速理解。但未來資料多時，可以拆成：

1. 總時間軸：保留現在的「九十五年」。
2. 專題歷史：如「嘉義管樂之都與嘉中」、「校友聯演四十年」、「樹人堂與文化局音樂廳」。
3. 考證資料：哪些節目冊、照片、海報支持哪些說法。

這樣可以避免 `history.html` 無限變長。

### 3.4 人物內容建議分層

人物應維持三層：

| 層級 | 用途 |
|---|---|
| 校友名錄 | 索引所有可公開校友 |
| 人物誌 | 精選有故事與代表性人物 |
| 個人專頁 | 完整人物資料 |

未來人物誌分類建議：

- 城市管樂推動者
- 指揮與團隊核心
- 職業演奏家
- 作曲與編曲
- 音樂教育者
- 校友聯演行政推手
- 社會各界知名校友
- 新世代校友

人物分類應允許一人多重角色。例如簡晟軒同時是長號演奏家、指揮、教師、校友聯演核心人物。

### 3.5 聯演內容建議分層

建議未來把校友聯演內容分成：

1. `concerts.html`：總覽與最新當屆。
2. `concerts/archive.html`：完整屆次年表與篩選。
3. `concerts/{year}-{nth}.html`：單屆詳細頁。
4. `data/concerts.js` 或未來來源資料：每屆結構化欄位。

單屆頁建議固定欄位：

- 屆次
- 年份
- 主題
- 日期
- 場地
- 主辦字頭
- 指揮
- 協奏者
- 曲目
- 演出人員
- 行政團隊
- 贊助或協辦
- 票務
- 相簿
- 海報
- 資料來源
- 資料狀態

### 3.6 相簿內容建議分層

影像館未來應從「照片展示」升級為「活動紀錄資料」。

每本相簿建議有：

- 相簿名稱
- 日期
- 活動類型
- 關聯屆次
- 關聯人物
- 場地
- 攝影者
- 精選照片
- 外部完整相簿連結
- 圖說來源
- 典藏狀態

這樣照片會成為歷史資料，不只是圖片。

## 4. 資料化與 AI 自動化建議

### 4.1 最終理想架構

建議長期目標是：

> 人只維護結構化內容，網站頁面由腳本產生。

可能架構如下：

```text
content/
  people/
    7581.md
    8861.md
  concerts/
    2013-29th.md
    2019-35th.md
  news/
    2026-07-02-weiwu-announce.md
  galleries/
    2026-weiwu.md
data/
  alumni-public.json
  number-lookup-public.json
  concerts.json
  site-config.json
templates/
  base.html
  person.html
  concert.html
  news.html
  gallery.html
scripts/
  build-site.js
  check-content.js
```

實際是否使用 `.md`、`.json`、`.yaml` 可以後續再決定。重點是內容與版型分離。

### 4.2 AI 新增最新消息的理想流程

未來可讓 AI 依固定欄位新增消息：

```yaml
date:
title:
summary:
category:
related_concert:
related_people:
thumb:
status:
body:
```

AI 產生後，自動流程應完成：

- 產生 `news/YYYY-MM-DD-slug.html` 或由 Markdown 生成 HTML。
- 更新 `data/news.js`。
- 更新 `feed.xml`。
- 更新 `sitemap.xml`。
- 檢查圖片路徑是否存在。
- 檢查是否有已建個人頁人物姓名需要連結。
- 檢查是否出現禁忌文案。

### 4.3 AI 新增人物的理想流程

建議人物資料欄位：

```yaml
num:
name:
year:
part:
tags:
roles:
profile_level:
photo:
public_summary:
people_card_summary:
roster_summary:
full_bio:
sources:
privacy_status:
review_status:
```

AI 或人工新增後，自動檢查：

- 編號格式是否合理。
- 民國 100 年後排序是否正確。
- 聲部代碼是否符合規則。
- 幹部是否有雙重聲部身份。
- 照片是否存在且為 WebP。
- 名錄簡述是否符合字數。
- 人物誌卡片是否符合字數。
- 是否需要新增 sitemap。
- 是否已記錄資料來源。
- 是否有本人確認或公開資料狀態。

### 4.4 AI 新增聯演資料的理想流程

建議單屆聯演欄位：

```yaml
nth:
year:
title:
date:
venue:
host_head:
conductors:
soloists:
program:
performers:
staff:
ticket:
poster:
gallery:
sources:
status:
notes:
```

檢查重點：

- 屆數是否與已知對照表衝突。
- 是否把推算資訊寫成確定。
- 是否誤寫 2026 首次售票或首次音樂廳。
- 場地名稱是否一致。
- 曲目是否標示暫定或正式。
- 是否同步總覽頁與獨立頁。

### 4.5 AI 自動化前必須先做的事

在大量讓 AI 寫內容前，必須先有：

1. 清楚的資料欄位格式。
2. 清楚的公開/內部資料邊界。
3. 清楚的資料狀態標籤。
4. 自動檢查腳本。
5. 修改歷程紀錄。

否則 AI 會加快產出，也會加快錯誤累積。

## 5. 視覺與使用體驗建議

### 5.1 不建議大改視覺風格

目前暖白、金色、古典字體、深色頁尾的方向適合網站內容。嘉義高中管樂隊暨校友管樂團的網站需要歷史感、可信度與情感，不適合改成過度潮流、過度商業或過度活動檔期式的設計。

建議保留：

- 暖白底色。
- 金色細節。
- Noto Serif TC 標題。
- Cormorant Garamond 的編號與年份氣質。
- 卡片、時間軸、表格、相簿的現有基礎元件。

### 5.2 建議加強大型內容閱讀

未來內容長度會增加，建議逐步補：

- 長人物頁的頁內目錄。
- 長聯演頁的資訊摘要卡。
- 文章頁的關聯人物與關聯屆次。
- 相簿頁的活動基本資料。
- 歷史專題的「資料狀態」標籤。

### 5.3 手機版重點

多數訪客可能先用手機瀏覽，因此手機版應優先呈現：

- 這是誰。
- 哪一年。
- 哪一屆。
- 在哪裡。
- 為什麼重要。
- 可以點去哪裡看更多。

長文可以保留，但頁首摘要與資訊卡必須足夠清楚。

### 5.4 表格策略

目前規範已決定手機版表格維持桌機比例並橫向捲動。這對名單、曲目、演出資訊是合理的。未來可補強：

- `.table-scroll` 加 `tabindex`。
- 加 `role="region"`。
- 加 `aria-label`。

這會提升鍵盤與螢幕閱讀器可用性。
