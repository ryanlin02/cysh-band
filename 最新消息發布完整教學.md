# 最新消息發布完整教學

建立時間：2026-07-11  
適用對象：完全不熟悉程式語言、HTML、JavaScript、Git 指令，但需要維護嘉義高中管樂隊暨校友管樂團官方網站最新消息的人。  
適用範圍：首頁最新消息、校友聯演頁最新消息、最新消息總覽、單篇最新消息文章、RSS、sitemap 與 GitHub Pages 發布前檢查。

---

## 1. 先講結論：你實際上要做什麼

發布一篇最新消息，不需要你理解網頁程式碼。你只需要把資料準備好，交給 Codex 或其他 AI 協作者依網站流程處理。

你要準備的是：

1. 這篇消息要講什麼。
2. 哪些資訊已經確認，哪些只是暫定。
3. 是否有圖片。
4. 圖片能不能公開使用。
5. 這篇消息是否需要置頂。
6. 發布後是否要立刻用 GitHub Desktop 推送。

AI 要負責的是：

1. 把正文放進 `content/news/`。
2. 把消息資料填進 `data/news.js`。
3. 產生正式文章頁、最新消息總覽與 RSS。
4. 更新 `sitemap.xml`。
5. 更新 `網站修改歷程紀錄.md`。
6. 執行網站檢查。
7. 回報你哪些檔案被改、檢查是否通過。

---

## 2. 最新消息現在的網站架構

目前最新消息是「資料驅動」流程。你可以把它想成三層：

| 層級 | 檔案位置 | 你可以怎麼理解 |
|---|---|---|
| 文章正文 | `content/news/` | 這裡放文章真正的內容，例如段落、圖片、表格。 |
| 文章資料卡 | `data/news.js` | 這裡放標題、日期、分類、標籤、摘要、是否置頂。 |
| 正式網站頁面 | `news/*.html`、`news/index.html`、`feed.xml` | 由產生器自動產生，不建議手動改。 |

重點：正式 `news/*.html` 不是主要編輯位置。  
如果要改文章，應該先改 `content/news/` 和 `data/news.js`，再讓產生器重做正式頁。

## 3. 發布一篇消息的完整流程

### 第 1 步：決定這篇消息的用途

先判斷它是哪一種消息。網站目前建議使用這些分類：

| 分類 | 適合內容 |
|---|---|
| 演出公告 | 音樂會日期、地點、售票、節目、正式演出資訊。 |
| 團練通知 | 團練時間、取消、改期、地點調整、集合提醒。 |
| 售票資訊 | OPENTIX、票價、購票連結、索票提醒。 |
| 活動紀錄 | 團練紀錄、烤肉、探班、校友回來、活動花絮。 |
| 相簿更新 | 影像館新增相簿、照片補檔、活動照片上線。 |
| 人物專訪 | 校友人物故事、指揮、獨奏者、在校生故事。 |
| 歷史補檔 | 舊聯演資料、節目冊、海報、校史資料補充。 |
| 行政公告 | 停班停課、防颱、安全提醒、網站維護、正式通知。 |

分類不要太多，否則未來總覽頁會變亂。  
一篇文章只能有一個主要分類，但可以有多個標籤。

---

### 第 2 步：決定標籤

標籤是用來把相關文章串起來的，不是單純裝飾。  
標籤可以比較彈性，但要真實、有用、不要亂加。

常用標籤範例：

| 標籤 | 適合用在 |
|---|---|
| 為伍 | 第 41 屆《為伍》相關消息。 |
| 校友聯演 | 歷屆或當屆聯演相關消息。 |
| 團練 | 團練紀錄、團練提醒。 |
| 團練通知 | 改期、取消、時間調整。 |
| 售票 | 票價、購票連結、售票提醒。 |
| OPENTIX | OPENTIX 售票相關消息。 |
| 文化局音樂廳 | 嘉義市政府文化局音樂廳相關消息。 |
| 停班停課 | 天災或市府公告。 |
| 巴威颱風 | 颱風特定事件。 |
| 校友歸隊 | 校友招募、校友回來團練。 |
| 在校生 | 在校生相關活動。 |
| 影像館 | 相簿、照片、影像網站更新。 |

建議每篇文章放 2 到 5 個標籤。  
不要每篇都塞一大堆標籤，熱門標籤區會失去意義。

---

### 第 3 步：判斷是否置頂

`pinned` 的意思是「重要公告」。  
首頁與校友聯演頁會優先顯示置頂消息。

適合置頂：

- 停班停課。
- 團練取消或改期。
- 售票正式開始。
- 音樂會重要異動。
- 網站重大公告。

不一定要置頂：

- 一般團練紀錄。
- 活動花絮。
- 相簿更新。
- 歷史補檔。

原則：同一時間最好只置頂 1 篇。  
如果每篇都置頂，置頂就失去功能。

---

### 第 4 步：整理素材

你可以先建立一個資料夾，例如：

```text
20260711_官網發布文章_主題名稱/
```

裡面放：

```text
01_參考資料/
02_可公開圖片/
03_不可公開參考截圖/
04_文章草稿/
```

這樣分資料夾的好處是，AI 比較不會把「只供參考的截圖」誤放進網站。

圖片請先分清楚：

| 類型 | 可不可以放網站 |
|---|---|
| 自己拍攝、可公開的照片 | 可以。 |
| 官方公告圖，來源清楚，適合轉載 | 通常可以，但要在圖說標示來源。 |
| 社群群組截圖 | 原則上不要放。 |
| 私人對話截圖 | 不要放。 |
| 含電話、地址、LINE、email、私人帳號的截圖 | 不要放。 |
| 未確認授權的照片 | 先不要放。 |

如果你只是想讓 AI 參考截圖內容，請明確說：

```text
這些截圖只供撰文參考，不要放到網站。
```

## 4. 一篇最新消息的建議內容架構

### 一般公告型

適合：售票、演出、團練、停班停課、行程調整。

建議順序：

1. 第一段：直接講重點。
2. 第二段：說明原因或背景。
3. 圖片或公告圖。
4. 條列重點。
5. 後續安排。
6. 資料來源或備註。

範例架構：

```text
第一段：嘉義市政府已公告 7 月 11 日停止上班上課，因此 7/11 團練取消。

第二段：7/12 是否照常，將依後續天候與學校安全狀況另行通知。

圖片：嘉義市政府公告圖。

重點：
- 7/11 團練取消。
- 7/12 暫待後續公告。
- 請大家注意安全。

結尾：請各聲部協助轉知，並以後續正式公告為準。
```

---

### 活動紀錄型

適合：團練紀錄、校友回來、活動花絮、烤肉、探班。

建議順序：

1. 第一段：時間、地點、這次活動是什麼。
2. 第二段：現場發生什麼，有什麼值得記錄。
3. 圖片與圖說。
4. 補充人物或團練狀況。
5. 下一步或後續時程。
6. 溫和結尾。

語氣可以有人味，但不要過度宣傳或像廣告。

---

### 演出公告型

適合：音樂會正式公告、售票、節目資訊。

建議順序：

1. 第一段：演出名稱、日期、時間、地點。
2. 第二段：這場演出的意義。
3. 售票或入場資訊。
4. 曲目或演出人員，若尚未確定要寫「以正式公告為準」。
5. 海報或主視覺。
6. 重要提醒。

注意：

- 2026《為伍》不要寫成首次在文化局音樂廳演出。
- 2026《為伍》不要寫成首次售票。
- 可寫「睽違六年重返嘉義市政府文化局音樂廳」。
- 可寫「透過 OPENTIX 售票」。

---

### 相簿更新型

適合：影像館新增相簿、照片整理完成。

建議順序：

1. 第一段：哪一個相簿已上線。
2. 第二段：這批照片記錄什麼。
3. 連到影像館或相簿。
4. 若有照片來源或拍攝者，簡要致謝。
5. 若會持續更新，寫清楚「持續更新中」。

## 6. 你不需要自己理解但要知道用途的檔案

### `content/news/`

這裡是文章正文。  
你可以把它想成「文章草稿的正式來源」。

### `data/news.js`

這裡是每篇消息的資料卡。  
網站首頁、校友聯演頁、最新消息總覽、RSS 都會讀這份資料。

你不用自己寫，但要知道每篇消息大概會有這些欄位：

| 欄位 | 白話意思 |
|---|---|
| `id` | 這篇文章的代號，通常跟檔名一樣。 |
| `date` | 發布日期。 |
| `time` | 發布時間。 |
| `category` | 文章分類。 |
| `tags` | 標籤。 |
| `pinned` | 是否置頂。 |
| `priority` | 重要程度。 |
| `title` | 列表上看到的標題。 |
| `summary` | 列表與 RSS 用的一句摘要。 |
| `source` | 正文來源位置。 |
| `output` | 正式文章頁位置。 |
| `thumb` | 列表縮圖。 |
| `headlineHtml` | 文章頁大標題。 |

### `news/*.html`

這是網站真的顯示給讀者看的文章頁。  
它是自動產生的，原則上不要直接手改。

### `news/index.html`

最新消息總覽頁。  
它會顯示所有消息、分類、熱門標籤和重要公告。

### `feed.xml`

RSS 訂閱檔。  
它現在由產生器自動更新，不需要手動改。

### `sitemap.xml`

給搜尋引擎看的網站地圖。  
新增文章時仍需要新增文章網址，Codex 會處理。

### `網站修改歷程紀錄.md`

每次實質修改網站都要記錄。  
這是日後追蹤「為什麼改、改了什麼、怎麼驗證」的地方。

## 7. 圖片處理規則

最新消息圖片建議分兩種：

| 類型 | 用途 | 建議尺寸 |
|---|---|---|
| 正文圖 | 放在文章內 | 長邊約 1200 到 1600 px |
| 縮圖 | 放在列表 | 約 480 px 寬 |

圖片格式以 WebP 為主。  
檔名請使用英文、數字、連字號，不要用中文。

好檔名：

```text
2026-07-10-chiayi-city-closure.webp
2026-07-10-chiayi-city-closure-thumb.webp
```

不建議：

```text
嘉義市停班停課圖片.webp
截圖 2026-07-10 下午9點.webp
```

## 8. 文案語氣規則

官方網站最新消息的語氣建議是：

- 清楚。
- 中性。
- 可以有溫度，但不要太活潑。
- 不要像廣告。
- 不要把未確認資訊寫成定論。
- 不要用過度華麗的形容詞掩蓋資料不足。

適合的句子：

```text
7 月 12 日（日）的團練是否照常，將依後續天候與學校安全狀況另行通知。
```

不適合的句子：

```text
颱風再大也吹不散我們對音樂的熱情！
```

原因：這種句子在安全公告中不夠穩重。

## 9. 不同消息類型的標題寫法

### 團練通知

好標題：

```text
因巴威颱風影響，7/11 團練取消
嘉義市 7/11 停止上班上課，請留意巴威颱風動態
```

特點：讀者一眼知道發生什麼事。

### 活動紀錄

好標題：

```text
7/4 團練日：火雞肉飯、團練室與一壺咖啡
《為伍》第一次團練啟動
```

特點：有記憶點，但仍清楚。

### 演出公告

好標題：

```text
第 41 屆聯合音樂會《為伍》8/8 文化局音樂廳登場
```

特點：演出名稱、日期、地點都清楚。


## 11. 發布到 GitHub Pages 的人工流程

網站檔案改好後，通常你會用 GitHub Desktop 發布。

簡化流程：

1. 打開 GitHub Desktop。
2. 確認左上角 repository 是 `cysh-band`。
3. 看 Changes 列表，確認有這次要發布的檔案。
4. 在左下角 Summary 寫簡短提交訊息，例如：

```text
新增最新消息：7/11 停班停課公告
```

5. 按 `Commit to main`。
6. 按 `Push origin`。
7. 等 GitHub Pages 自動更新。
8. 到 `https://cysh.band/` 檢查首頁與最新消息頁。

發布前如果 Codex 說檢查沒有通過，不要先推送。

## 12. 發布後檢查

推送後建議看這些地方：

1. 首頁：重要公告和最新消息是否正常。
2. 校友聯演頁：最新消息是否正常。
3. 最新消息總覽：分類與熱門標籤是否正常。
4. 單篇文章：圖片是否正常、文字是否正確。
5. 手機版：標題、圖片、表格是否沒有擠出畫面。

如果是緊急公告，例如停班停課，建議另外檢查：

1. 日期是否正確。
2. 星期是否正確。
3. 是否清楚寫出取消或改期。
4. 是否有「以後續正式公告為準」。
5. 是否沒有放進私人截圖或群組對話。

## 13. 常見錯誤

### 錯誤 1：只改了文章頁，沒有改資料檔

結果：首頁與總覽頁不會出現新文章。  
正確做法：要更新 `data/news.js`，再跑產生器。

### 錯誤 2：只改 `data/news.js`，沒有跑產生器

結果：正式文章頁、總覽頁或 RSS 可能沒有更新。  
正確做法：跑 `node scripts/generate-news-pages.js`。

### 錯誤 3：把參考截圖放到網站

結果：可能公開私人對話或個資。  
正確做法：明確分成「可公開圖片」與「只供參考截圖」。

### 錯誤 4：同時置頂太多篇

結果：首頁會失去重點。  
正確做法：同一時間通常只置頂 1 篇最重要消息。

### 錯誤 5：標籤太亂

結果：熱門標籤無法幫讀者找文章。  
正確做法：用穩定、可重複使用的標籤。

## 14. 最推薦的日常工作方式

最穩定的方式是：

1. 你先整理素材和重點。
2. 你把素材路徑、文章重點、哪些圖可公開、哪些圖不可公開寫清楚。
3. 交給 Codex 依本教學新增最新消息。
4. Codex 修改檔案並執行檢查。
5. 你用 GitHub Desktop 推送。
6. 你打開正式網站做最後確認。

對不熟程式的人來說，這比自己手動改檔案安全很多。

## 15. 給 Codex 的固定任務檢查

每次你請 Codex 發最新消息時，可以加上這句：

```text
請依照《最新消息發布完整教學.md》的流程處理，完成後回報修改檔案、檢查指令與是否通過。若有任何圖片或文字可能涉及個資、授權或不確定資訊，請先指出並保守處理。
```

這樣可以讓後續 AI 協作者直接按照同一套規則工作。
