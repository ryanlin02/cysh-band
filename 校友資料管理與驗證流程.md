# 嘉義高中管樂隊官方網站 校友資料管理與驗證流程

建立時間：2026-07-04 22:13 CST（Asia/Taipei，UTC+8）  
建立者：Codex（依使用者補充情境整理）  
文件目的：釐清 `data/alumni.js`、`data/number-lookup.js`、個人介紹頁、人物誌與內部 Excel 名冊之間的角色分工，建立一套不會誤公開個資、也能逐步補齊資料的維護流程。

---

## 1. 核心結論

目前不建議把內部 Excel 名冊直接轉成網站前端資料庫，也不建議把 `data/number-lookup.js` 自動併入 `data/alumni.js`。

較安全、可長期維護的做法是：

1. `data/alumni.js` 繼續作為「公開校友名錄」資料來源。
2. `people/{編號}.html` 作為單一人物最完整的公開介紹頁。
3. `people.html` 作為「人物誌」策展頁，只挑選代表性人物。
4. `data/number-lookup.js` 只作為「編號小遊戲」資料，不視為已驗證公開名錄。
5. Excel 名冊只作為 repo 外的內部參考資料，絕不整份上傳 repo。
6. 所有資料從內部來源進入公開網站前，都必須經過「公開性」與「正確性」兩層判斷。

## 2. 各資料檔角色

### 2.1 `data/alumni.js`：公開校友名錄

定位：公開網站上可以顯示的校友基本資料。

用途：

- 驅動 `roster.html` 校友名錄。
- 提供人物頁連結。
- 未來可作為所有公開人物資料的索引。

資料要求：

- 姓名、編號、入學年、聲部、簡短介紹皆應可公開。
- 不放電話、地址、email、私人社群帳號、內部備註。
- 可以先有基本資料，不一定要立即有個人頁。
- 若資料還不完整，應以保守簡短文字呈現。

未來可逐步加上的狀態欄位：

| 欄位 | 建議值 | 用途 |
|---|---|---|
| `profileStatus` | `none`、`draft`、`published`、`featured` | 是否已有個人頁、是否進人物誌 |
| `verificationStatus` | `verified`、`self-reported`、`inferred-from-number`、`legacy-roster`、`needs-review` | 資料正確性狀態 |
| `privacyStatus` | `approved-public`、`pending-confirmation`、`internal-only` | 是否可公開 |
| `sourceNotes` | 簡短來源說明 | 供維護者追蹤，不放敏感資料 |

短期不必一次改完所有資料結構；可先在文件中定義，等下一輪名錄整理時逐步加欄。

### 2.2 `people/{編號}.html`：個人最完整介紹頁

定位：單一人物最完整、最可讀、最具脈絡的公開介紹。

用途：

- 放完整履歷、故事、參與聯演紀錄、代表作品、公開來源。
- 未來所有名錄卡、人物誌卡片都應從這裡萃取摘要。
- 已有個人頁者，`data/alumni.js` 的 `link` 應指向個人頁。

維護原則：

- 先做已出現在人物誌的人。
- 再做聯演核心人物、已確認資料較完整者。
- 不為了追求數量而建立內容不足或未確認的頁面。

### 2.3 `people.html`：人物誌策展頁

定位：精選人物，不是完整名錄。

收錄條件：

- 社團、音樂界、校友團、嘉義管樂發展或社會各界有代表性。
- 具有足夠可公開介紹的故事或資料。
- 不必等同於所有已建立個人頁者。

文字來源：

- 人物誌卡片應從個人頁內容濃縮。
- 字數比名錄卡多，但仍要符合 `網站製作規範.md` 的卡片字數規範。

### 2.4 `data/number-lookup.js`：編號小遊戲資料

定位：互動查號用資料，不是完整公開名錄，也不是權威資料庫。

用途：

- 讓使用者輸入或轉動編號後看到對應姓名與聲部。
- 展示嘉中管樂編號文化。

限制：

- 內容未完全驗證。
- 不可自動併入 `data/alumni.js`。
- 不可因為存在於查號資料，就自動建立個人頁或人物誌卡片。

可作為候選資料的情境：

- 若 `data/alumni.js` 某人缺聲部，而 `data/number-lookup.js` 有同編號聲部，可列為「候選聲部」。
- 編號第三碼本身可推導聲部，但幹部編號第三碼為 0 時仍需查明實際聲部。
- 候選資料必須人工確認後才寫入公開名錄。

### 2.5 內部 Excel 名冊：repo 外參考資料

定位：內部整理與交叉比對來源，不是網站公開資料。

本次檢視到的工作表：

| 工作表 | 性質 | 初步判斷 |
|---|---|---|
| `校友聯絡冊` | 聯絡資料與現職資料 | 含地址、電話、email，不能進 repo |
| `管樂隊成員名冊` | 成員聯絡資料 | 含手機、地址、email，不能進 repo |
| `舊嘉中校友名冊` | 舊版名冊，部分有聲部代碼 | 可作為候選來源，但仍需人工判斷 |
| `CD目錄_CB` | CD 曲目目錄 | 與校友公開名錄無直接關係 |

本次檢視摘要：

- `校友聯絡冊`：464 筆可辨識編號，與 `data/number-lookup.js` 高度重疊。
- `管樂隊成員名冊`：193 筆可辨識編號。
- `舊嘉中校友名冊`：130 筆可辨識編號，其中聲部欄較有比對價值。
- 這份 Excel 對「聯絡資訊」價值高，但對目前 `data/alumni.js` 缺聲部的 50 筆資料，沒有直接提供可套用的聲部候選。

重要原則：

- 不將 Excel 原檔放入網站 repo。
- 不將電話、地址、email、私人社群資料轉成前端 JS。
- 可用 Excel 人工比對，但只把確認後、可公開的結果寫入 `data/alumni.js` 或個人頁。

## 3. 建議資料狀態分級

### 3.1 正確性狀態

| 狀態 | 意義 | 可否公開 |
|---|---|---|
| `verified` | 已由本人、公開來源、正式節目冊或可信資料確認 | 可以 |
| `self-reported` | 本人填寫或直接提供 | 通常可以，但仍需確認公開同意 |
| `inferred-from-number` | 由編號第三碼推導聲部 | 可作候選，寫入前需人工看過 |
| `legacy-roster` | 來自舊名冊或內部歷史檔 | 可作候選，不自動公開 |
| `needs-review` | 資料有疑問或來源不明 | 不建議公開新增 |

### 3.2 公開狀態

| 狀態 | 意義 | 處理方式 |
|---|---|---|
| `approved-public` | 已確認可公開 | 可進 `data/alumni.js`、個人頁或人物誌 |
| `pending-confirmation` | 資料可能正確，但尚未確認是否可公開 | 先不要新增到公開名錄 |
| `internal-only` | 只供內部聯絡或考證 | 不進 repo，不進前端 JS |

### 3.3 個人頁狀態

| 狀態 | 意義 |
|---|---|
| `none` | 只有名錄資料，沒有個人頁 |
| `draft` | 正在整理個人頁，尚未公開 |
| `published` | 已有公開個人頁 |
| `featured` | 已進人物誌策展頁 |

## 4. 建議工作流程

### 4.1 新增或更新一位校友

1. 收集來源：本人提供、公開資料、節目冊、聯演名單、內部 Excel、舊名冊、查號資料。
2. 判斷是否可公開：
   - 可以公開：進下一步。
   - 不確定：標記為待確認，不寫入公開資料。
   - 明顯內部資料：留在 repo 外。
3. 判斷正確性：
   - 姓名、編號、入學年、聲部是否一致。
   - 若只靠編號推導聲部，標記為候選，不直接當作完全驗證。
4. 更新 `data/alumni.js`：
   - 只放公開基本資料。
   - `desc` 保持短摘要。
   - 有個人頁才填 `link`。
5. 若資料足夠，建立 `people/{編號}.html`。
6. 若具代表性，再加入 `people.html` 人物誌。
7. 更新 `網站修改歷程紀錄.md`。
8. 執行 `node scripts/check-site.js`。

### 4.2 從 `data/number-lookup.js` 補聲部到 `data/alumni.js`

建議採「候選清單 → 人工確認 → 分批寫入」流程。

目前觀察：

- `data/alumni.js` 有 50 筆聲部或 tags 不完整。
- 這 50 筆在 `data/number-lookup.js` 都有候選聲部。
- 其中 48 筆可由編號第三碼直接推導聲部。
- 2 筆幹部編號第三碼為 0，需要額外確認實際聲部。

建議做法：

1. 先由 Codex 產生「公開資料候選清單」，只使用 `data/alumni.js` 與 `data/number-lookup.js`。
2. 你人工看過清單，確認哪些可以寫入。
3. 第一批只補「非幹部、編號第三碼可直接推導且 lookup 一致」者。
4. 幹部組與姓名異動者另外確認。
5. 每批更新後執行 `node scripts/check-site.js`。

### 4.3 從 Excel 名冊補現職或其他介紹

Excel 可作為候選來源，但不能直接轉入網站。

建議流程：

1. 從 Excel 只讀取編號、姓名、現職等非聯絡欄位作為候選。
2. 不讀取或輸出電話、地址、email。
3. 若現職資訊過舊，需本人或公開資料再確認。
4. 若只是名錄卡，可寫成保守摘要。
5. 若要寫入個人頁，應列出公開來源或註明資料由本人提供。

## 5. 空白照片處理建議

目前 `data/alumni.js` 中仍有大量 `photo: "blank"`。

使用各聲部樂器特寫作為替代圖是可行的，但建議不要使用寫實照片直接混同真人大頭照，而是製作一套風格一致的「聲部預設圖」。

建議做法：

1. 產生或設計 9 張聲部預設圖：
   - 長笛、豎笛、薩克斯風、法國號、小號、長號、上低音號、低音號、打擊。
2. 放在：
   - `assets/img/members/placeholders/flute.webp`
   - `assets/img/members/placeholders/clarinet.webp`
   - 依此類推。
3. 修改名錄渲染邏輯：
   - 若 `photo !== "blank"`，使用個人大頭照。
   - 若 `photo === "blank"` 且有明確聲部，使用聲部預設圖。
   - 若聲部不明，才使用社徽 `blank.webp`。
4. 視覺上應讓讀者知道這是預設圖，不是真人照片。

建議暫不立即實作，等聲部資料補齊後再一次製作，效果會更好。

## 6. 建議下一步

依目前狀況，建議下一步不是改 Excel，也不是大改資料結構，而是先做一個低風險資料整理階段：

1. 產生 `data/alumni.js` 缺聲部的候選清單。
2. 只使用公開 repo 內資料：`data/alumni.js`、`data/number-lookup.js` 與編號規則。
3. 將候選分成：
   - 可由編號第三碼與 lookup 一致推定者。
   - 幹部組需人工確認者。
   - 姓名或資料有差異需人工確認者。
4. 你確認後，再分批寫入 `data/alumni.js`。

這樣做的好處是：

- 不會碰內部個資。
- 不會把未驗證 Excel 內容直接公開。
- 可以先解決名錄篩選功能中最明顯的缺漏。
- 未來若要製作聲部預設圖，也會有更可靠的聲部分類。
