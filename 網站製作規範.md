## 一、網站定位與資訊架構

### 1.1 定位
以「讓大家了解嘉義高中管樂隊全貌」為核心的樂團形象網站：年輕隊員往回看歷史，資深校友往前看新世代。音樂會宣傳（如《為伍》）是階段性任務，網站是長期資產，每年聯演後持續累積內容。

### 1.2 導覽架構（8 項扁平）
首頁｜關於樂團｜九十五年｜編號｜人物誌｜校友名錄｜校友聯演｜影像館

- 採扁平結構，每頁一次點擊直達；8 項為上限，再增加頁面時應改為下拉分群，並重新評估
- 行動版（≤900px）收合為漢堡選單
- 每頁導覽列完全一致，僅 `class="active"` 隨頁面變動；`news/` 內文章頁連結需加 `../` 前綴，active 固定在「校友聯演」

### 1.3 頁面職責分工
| 頁面 | 職責 | 成長方式 |
|---|---|---|
| index.html | 導流樞紐：雙向敘事入口＋當屆音樂會＋最新消息 | 每年換當屆資訊 |
| about.html | 團隊簡介、立案資料 | 少變動 |
| history.html | 歷史時間軸（1931→今） | 偶爾補時間軸節點 |
| numbers.html | 編號文化敘事（全站靈魂頁） | 少變動，例子維持精選 |
| people.html | 精選人物故事（策展性質） | 只收「有故事可講」者 |
| roster.html | 完整校友名錄（資料驅動） | 主要成長區，加 JSON 即可 |
| concerts.html | 最新消息＋當屆演出＋歷屆紀錄 | 每年歸檔一屆、發消息 |
| gallery.html | 活動相簿 | 逐相簿增加 |

人物誌與名錄允許重複收錄同一人：名錄是完整索引、人物誌是精選故事，名錄卡以 `link` 欄位連往人物誌。

### 1.4 目前架構狀態與長期方向（新增，2026-07-04）
截至 2026-07-04，本站仍是**純靜態網站**：HTML／CSS／原生 JavaScript 直接由 GitHub Pages 部署，沒有建置工具、資料庫、後台或前端框架。此架構穩定、輕量、容易部署，短期內不應為了重構而重構。

但網站內容已逐漸具備「長期知識庫」性質，包含人物、校友名錄、歷屆聯演、歷史年表、相簿、最新消息、編號文化與考證資料。未來擴張時的長期方向是：

1. **保留現有視覺風格與 GitHub Pages 靜態部署**。
2. **逐步將大量內容改為資料驅動**，優先處理人物、聯演、消息與相簿。
3. **逐步模板化共用區塊**，包含 nav、footer、head、SEO、sitemap、feed。
4. **建立自動檢查流程**，讓 AI 或人工新增內容後可以檢查連結、圖片、資料格式、禁忌文案與 SEO 同步。
5. **所有架構調整必須分階段紀錄於 `網站修改歷程紀錄.md`**，不可只改檔案而不留下維護脈絡。

`網站結構優化建議報告書.md` 是本方向的第 1 次完整評估，後續若進行資料化或模板化，應先對照該報告的分階段建議。

### 1.5 共用模板試作流程（新增，2026-07-04）
截至 2026-07-04，已建立 `templates/` 作為共用模板試作區，目標是先把每頁重複的 `head`、導覽列與頁尾集中管理，再逐步評估是否套用到正式頁面。

目前規則：

1. `templates/partials/head.html`、`nav.html`、`footer.html` 與 `templates/layouts/base.html` 是模板原型。
2. `scripts/generate-page-preview.js` 只輸出 `_generated/page-template-preview.html`，不得直接覆蓋任何正式頁面。
3. `_generated/` 內檔案屬於本地預覽與比對用途，不加入 `sitemap.xml`。
4. 正式頁面要套用模板時，應分批處理；優先從最新消息文章、人物個人頁、相簿頁或聯演屆別頁等格式規律的頁面開始。
5. 每次模板化正式頁面前後都必須執行 `node scripts/check-site.js`，並在 `網站修改歷程紀錄.md` 留下紀錄。

第一批正式頁面模板化試點已擴充為目前全部 5 篇最新消息文章：

- `news/2026-06-12-rehearsal-schedule.html`
- `news/2026-06-27-first-rehearsal.html`
- `news/2026-06-30-summer-bbq.html`
- `news/2026-07-02-weiwu-announce.html`
- `news/2026-07-04-rehearsal-coffee.html`

這些文章的正文來源改放在 `content/news/`，並由 `scripts/generate-news-pages.js` 產生正式 HTML。GitHub Pages 不會自動執行 Node 腳本，因此每次修改 `content/news/` 後，都必須在本機執行 `node scripts/generate-news-pages.js`，確認輸出的 `news/*.html` 已同步，再 commit / push。

`news/_template.html` 已改為模板化流程說明，不再作為複製整頁 HTML 的範本；新增消息時應依 README 的模板化流程操作。`node scripts/check-site.js` 會比對 `scripts/generate-news-pages.js` 產生的 HTML 與目前正式 `news/*.html`，若兩者不同，代表忘記重跑產生腳本或產生結果未提交。

短期內不建議直接全面導入 React、Vue、Astro、Eleventy 或其他大型框架；本站仍以 GitHub Pages 可直接部署的靜態 HTML 為核心，模板化只作為降低重複維護的輔助工具。

人物個人頁是下一個模板化候選區域。正式轉換 `people/*.html` 前，必須先參考 `人物頁模板化規格與檢查清單.md`，確認頁面標準區塊、資料來源分級、相關校友聯演區塊、相關連結區塊與健康檢查項目，再採小批次試作，不可直接一次覆蓋全部人物頁。

`scripts/check-site.js` 已加入人物個人頁基本結構檢查，會確認 `people/*.html` 具備頁首人物區、正文、資料來源、人物頁導覽、頁首圖片與 `data/alumni.js` 對應連結。此檢查只驗證結構，不判斷人物成就、來源可信度或完整聯演履歷。

## 二、視覺設計系統

### 2.1 色彩（css/style.css `:root` 為唯一定義處，禁止硬寫色碼）
| 變數 | 色值 | 用途 |
|---|---|---|
| --bg | #faf8f3 | 全站底色（暖白紙質） |
| --bg-soft | #f2eee4 | 區塊底、數字帶 |
| --bg-card | #ffffff | 卡片底 |
| --ink | #2b2721 | 主文字 |
| --ink-dim | #6f6757 | 次要文字 |
| --gold | #a07c31 | 主金（連結、強調，深色以保淺底對比） |
| --gold-soft | #c8a45c | 亮金（漸層、hover、深色區塊上的金） |
| --line | #e6dfd0 | 邊框、分隔線 |
| --dark / --dark-2 | #17140f / #221c14 | 深色區塊（音樂會 banner、頁尾） |
| --dark-ink / --dark-ink-dim | #ece5d8 / #b3a892 | 深色區塊上的文字 |

設計原則：全站淺色，深色僅保留**頁尾**收尾。當屆音樂會區塊（.concert-banner、.featured）採**亮面米白漸層卡片**（白→#f4efe3、頂部 3px 金色飾線）——深色海報放在淺色底上對比最強、最能跳出，**禁止**再改回深色底（曾因深色底與深色海報互融而勘誤）。海報一律加金色細框＋呼吸光暈（.poster-wrap）。

### 2.2 字型（Google Fonts）
- 中文標題：Noto Serif TC（700／900）
- 中文內文：Noto Sans TC（400／700）
- 拉丁字與數字裝飾（kicker、編號、年份、統計數字）：Cormorant Garamond（500／600）
- 內文 16px、行高 1.85–1.9；標題字距 .1em 上下；英文 kicker 字距 .3–.55em

### 2.2-1 設計 token、互動與可及性（新增，2026-07-05）
`css/style.css` 的 `:root` 是全站設計 token 的主要來源。新增或調整元件時，應優先使用既有 token，不直接在元件內重複硬寫版面寬度、圓角、動畫時間、緩動曲線或 focus 陰影。

目前基準如下：

- 版面：`--layout-max` 控制主要內容最大寬度；`--page-x` 控制頁面左右安全留白。
- 圓角：`--radius-xs`、`--radius-sm`、`--radius-md`、`--radius-lg`、`--radius-xl`、`--radius-pill` 依元件大小使用。卡片、按鈕、篩選膠囊與相簿項目不另造新的圓角數值。
- 動畫：`--duration-fast`、`--duration-base`、`--duration-slow` 與 `--ease-standard` 為全站互動動畫基準；hover 位移使用 `--lift-sm`、`--lift-md`、`--lift-lg`。
- 可及性：`--shadow-focus` 是鍵盤 focus 的主要陰影。所有可點擊元件若新增樣式，必須確認 `:focus-visible` 狀態清楚可見。
- 減少動態：所有動畫與 transition 必須尊重 `prefers-reduced-motion`。新效果不得只依靠動態才能理解內容。

設計原則：互動效果以「輕微、穩定、能提示可點擊」為主；不要新增大幅度彈跳、旋轉、閃爍或會干擾閱讀的持續動畫。若需要更強烈的動態效果，應先建立成可重複使用的 class 或 token，再套用到特定元件。

次要元件整理原則（新增，2026-07-05）：卡片、表格、相簿、聯演項目、編號小遊戲、lightbox、人物頁頭像、篩選按鈕與手機選單等元件，凡屬於「元件外框圓角、hover 位移、transition 時間、緩動曲線」者，應優先使用上述 token。少數幾何用途的值可以保留硬寫，例如圓形頭像／圓點的 `50%`、漢堡選單線條厚度與叉叉位移、進度條 `.1s linear`，因為它們不是一般元件設計語言，而是形狀或功能本身。

### 2.3 元件庫（新內容一律沿用既有 class，不另造樣式）
- `.section`＋`h2`：標準內容區塊（h2 自帶金色橫線前綴）
- `.card`／`.card-head`：人物卡（頭像 104px 與編號姓名並排）。**若該人已有個人專頁**（新增，2026-07-04）：`.card-head` 改用 `<a class="card-head" href="people/{編號}.html" aria-label="查看{姓名}完整人物誌 →">` 包住頭像＋編號＋姓名，使三者皆可點擊直達個人專頁；對應樣式為 `a.card-head`（見 style.css，非另建 class）。尚未建立個人專頁者（如蕭萬長、伍佰等）維持原本 `<div class="card-head">`，不可點擊。此連結與卡片內文字（`<p>` 簡介、`更多` 連結）為同層級手足元素、不可互相巢狀，以避免 `<a>` 巢狀 `<a>` 的 HTML 錯誤
- `.cards`：卡片網格（auto-fill, minmax(280px,1fr)）
- `.concert-item`（＋`.has-poster`）：歷屆聯演列表項
- `.timeline`／`.tl-item`：歷史時間軸
- `.news-list`／`.news-item`：消息列表
- `.gallery-grid`＋`figure`：相簿（自動掛 lightbox）
- `.btn`／`.btn.ghost`：主要／次要按鈕
- `.filter-bar`／`.filter-btn`：篩選膠囊按鈕
- `.table-scroll > table.plain`：表格一律包防溢出容器
- `blockquote`：金色左線引言

### 2.4 動態效果（js/main.js 自動套用，新頁面只需掛 script）
捲動漸入（.section、.card、.tl-item 等自動偵測、群組交錯延遲）、統計數字滾動、導覽列捲動陰影、閱讀進度條、回頂按鈕、漢堡選單、lightbox、**音樂會區塊金色光澤掃過（sheen，7 秒週期）**、**演出倒數徽章**（元素加 `class="countdown" data-countdown="YYYY-MM-DD"` 即自動顯示「距離演出還有 N 天」，過期自動改為謝幕文案；每年換屆只需改日期）。所有動畫必須尊重 `prefers-reduced-motion`，且以「緩慢、細微、有目的」為原則，禁止持續晃動或干擾閱讀的效果。

## 三、響應式規範

| 斷點 | 行為 |
|---|---|
| ≤900px | 導覽收合為漢堡選單（觸控目標 ≥44px） |
| ≤760px | 音樂會深色區塊收單欄；歷屆聯演項（`.has-poster`）改為「海報縱貫左側＋屆數／內文疊在右側」；屆數改行內靠左 |
| ≤720px | 雙向入口收單欄 |
| ≤640px | 頁首／區塊留白縮減 |
| ≤560px | 消息項換行、隱藏箭頭 |
| ≤420px | 卡片頭像縮至 84px |

原則：同類元件必須在**同一斷點**切換版型（避免有海報／無海報各自為政的混亂）；卡片網格用 auto-fill 自動適應，不另設斷點；表格永遠包 `.table-scroll`。新增元件時先在 375px、768px、1080px 三個寬度檢查。

- **手機版表格規則（修訂，2026-07-04，二次修訂）**：全站表格（人物頁「基本資料」、歷屆聯演「演出資訊」「演出人員名單」等）在小螢幕下**一律維持與桌機相同的欄位比例／排版**，不壓縮欄寬、不轉為堆疊卡片；超出螢幕寬度時單純交由 `.table-scroll` 橫向捲動。
  - 曾經在 ≤640px 斷點把表格轉為「每列一張卡片」（th 標籤在上、值在下）；後來發現即使是這種堆疊也會讓欄寬被手機斷點強制縮小、內文被迫多次換行，觀感與桌機版落差太大，因此改為**完全不做手機專屬的表格覆寫**——`table.plain`／`table.plain.roster` 在所有斷點套用同一份桌機版規則，只靠 `.table-scroll { overflow-x:auto }` 因應窄螢幕
  - **例外（永久獨立呈現）**：「編號」頁（numbers.html）的聲部代碼表 `.part-code-table` 完全由其專屬 CSS 規則（非媒體查詢、任何斷點皆固定）控制外觀，日後對全站表格樣式的任何調整都不應影響到它；如需調整這張表，必須直接修改 `.part-code-table` 專屬規則，不要動到共用的 `table.plain` 規則
  - 新增表格時：一般表格直接沿用 `table.plain`（欄少直接用、欄多或名單類加 `.roster`）＋外層 `.table-scroll`，不需要額外處理手機版；只有像 `.part-code-table` 這種有特殊欄寬需求、且明確要「永久獨立、不受未來全站表格調整影響」的頁面才需要新增專屬 class 並比照隔離
  - **橫向捲動視覺提示（新增，2026-07-04）**：`.table-scroll`（`.part-code-table-wrap` 除外）已加上 CSS-only 的「捲動陰影」——表格超出可視寬度時，左右邊緣會自動出現淡出漸層＋輕微陰影提示可以滑動，捲到底／頂時該側自動消失，內容本來就沒有超出寬度則完全不出現，不需要 JavaScript 偵測。新增表格若沿用 `table.plain`＋外層 `.table-scroll` 即會自動套用，不需額外處理
  - 待優化方向（尚未實作，供後續參考）：`.table-scroll` 目前沒有 `tabindex`／`role="region"`／`aria-label`，鍵盤與螢幕報讀使用者暫時無法聚焦捲動；未來若要加強無障礙可用性可考慮補上

- **場次卡片（`.session`）手機版留白被覆蓋的問題（修正，2026-07-04）**：`concerts/2019-35th.html` 這類多場次介紹頁，場次區塊同時掛 `class="section session"`。原本 `.session` 在 `≤640px` 的 `padding` 覆寫規則，與後方「小螢幕微調」區塊裡權重相同的 `.section{padding:44px 0}` 發生**選擇器優先權相同、後到者覆蓋前者**的 CSS 疊層問題，導致手機版場次卡片橫向留白被歸零、內文緊貼卡片邊框。已將 `.session` 涉及留白／裝飾線位置的規則一律改寫為 `.section.session`（提高權重，不受後面規則的先後順序影響）。**通用原則**：任何元素若會同時掛 `.section` 與其他自訂 class，且該 class 也定義了會被 `.section` 覆寫的同名屬性（如 `padding`），一律用 `.section.xxx` 限定選擇器書寫，不要單獨寫 `.xxx { ... }`，避免日後又因為規則先後順序被悄悄蓋掉
- **歷屆聯演項（`.concert-item.has-poster`）手機版排法重新設計（修正，2026-07-04）**：原本手機版把「海報＋屆數」排成一列、「標題＋簡介」另起一列滿版，會讓海報看起來只跟屆數編號有關聯、跟真正的標題／內文脫節，视覺上「怪怪的」。已改為海報縱貫左側（`grid-template-areas: "poster no" "poster content"`），屆數與標題／簡介一起排在海報右側、上下堆疊，海報與整則紀錄的關聯更直覺

## 五、資料驅動內容

### 5.1 校友名錄（data/alumni.js → roster.html）
- 欄位：`num`（編號，無則 null）、`name`、`year`（入學民國年）、`part`（聲部顯示字）、`tags`（篩選標籤陣列，可複選如 ["幹部","低音號"]）、`role`、`desc`、`photo`（檔名不含副檔名）、`link`（人物誌連結，無則空字串）
- 定位：`data/alumni.js` 是**公開校友名錄與人物資料的主要來源**。它收錄的是目前可以公開刊登的校友基本資料，並供 `roster.html` 顯示；未來每位可公開校友原則上都可以逐步建立個人介紹頁，人物誌則從其中挑選具代表性或資料較完整者策展呈現。
- 聲部代碼（曾勘誤，4 與 5 勿再寫反）：0 幹部（01 社長、02 副社長、03 前幹部）、1 長笛、2 豎笛、3 薩克斯風、**4 法國號、5 小號**、6 長號、7 上低音號、8 低音號、9 打擊
- **幹部雙重身分**：第三碼為 0 的幹部同時隸屬某個聲部，資料一經查明一律補上——`part` 寫「職稱．聲部」（如「副社長．長號」）、`tags` 雙標籤（如 `["幹部","長號"]`），聲部篩選時兩邊都會出現
- 民國 100 年後入學者，年份碼取末兩碼（如 104 年→「04」開頭）
- 排序與分組由 js/roster.js 自動處理（雙檢視：預設「依字頭」＝編號第二碼分組；可切換「依入學年代」），資料檔內順序不拘
- 資訊不足的校友：只寫姓名、編號、入學年、聲部即可，desc 用「民國 XX 年入學，XX 聲部校友。」格式
- **名錄卡簡述原則**：desc 以 60 字（約兩行）為上限，維持卡片牆版面整齊；經歷豐富者名錄卡只留一句話摘要，完整介紹放人物誌卡片，故事量大者再開個人專頁（見 7.3），`link` 指向對應頁面
- **各層級文字量規範（新增，2026-07-04）**：全站同一人在不同頁面的介紹文字長度，依所屬層級固定如下，目的是讓 roster.html 名錄卡與 people.html 人物誌卡片牆版面在大量增補內容後仍維持整齊：
  1. **個人專頁**（`people/{編號}.html`）：不限字數，盡可能詳盡介紹該人物所有可查證資訊。
  2. **人物誌卡片**（`data/people-profiles.js` 的 `PEOPLE_FEATURED_SECTIONS[].items[].summaryHtml`，由 `scripts/generate-people-index.js` 產生到 people.html）：無論是否有個人專頁，一律控制在約 120 個可視字元上下，以維持整個頁面卡片高度一致、版面統一美觀。**字數區間收斂（修訂，2026-07-05）**：因卡片摘要常包含年份、英文團名、曲名與內部連結，實務上同時檢查「可視總長」與「中文主體長度」；`node scripts/check-site.js` 目前警示範圍為可視總長 105–145 字、中文主體 75–115 字。資訊不足者以本人個人專頁既有內容或可查證公開資料擴寫，資訊過多者精簡至區間內；新增校友卡片時也應對照此區間撰寫，不可只套用「約 120 字」而放任偏短或偏長。
  3. **校友名錄卡片**（data/alumni.js `desc` 欄位）：一律控制在約 40 個中文字左右（見上方「名錄卡簡述原則」，此為更嚴格的細項規定，40 字取代原先 60 字上限）。真的沒有特殊事蹟者，40 字以內簡單呈現職業／頭銜／入學年與聲部即可，也可以呈現近期參與過哪幾屆校友聯演。
  - 中文字數計算僅計 CJK 漢字本體，不計標點符號、英數字與空格。
- **people.html 收錄門檻（新增，2026-07-04）**：目前僅收錄在社團內或音樂界表現傑出者。未來規劃將新增在社會各界（不限音樂領域）表現傑出的知名校友，屆時比照現行卡片格式與字數規範新增分類區塊即可，不需另建頁面架構。
- **「有個人專頁但不進人物誌」情境（新增，2026-07-04）**：部分校友雖已建立個人專頁，但尚未有足以收錄進 people.html 策展名單的突出事蹟（例如仍在學、聯演參與尚淺）。此類校友僅於校友名錄（roster.html／data/alumni.js）呈現，`desc` 依上方名錄卡規範控制在約 40 字，`link` 指向其個人專頁（`people/{編號}.html`）供讀者點閱完整介紹；待未來事蹟足以策展收錄時，再補上 people.html 卡片並將其歸類至合適分區。範例：陳羿弦（0741，法國號，現就讀中山大學藝術管理與創業研究所）。
- **名錄卡片連結行為（新增，2026-07-05）**：若 `data/alumni.js` 的 `link` 指向 `people/{編號}.html`，`roster.html` 動態產生的名錄卡會將頭像、編號與姓名包成 `<a class="card-head">`，點擊即可進入個人頁；沒有個人頁者維持純展示。此規則比照人物誌卡片，但仍保留卡片底部「詳細介紹 →」文字連結，兩者不可巢狀。
- **人物誌首頁資料化（新增，2026-07-05）**：`people.html` 不再直接手改卡片 HTML；卡片分區、角色、摘要與少數無個人頁卡片資料，統一放在 `data/people-profiles.js` 的 `PEOPLE_FEATURED_SECTIONS`。修改後須執行 `node scripts/generate-people-index.js` 重新產生 `people.html`，再執行 `node scripts/check-site.js` 確認同步。若人物已有個人頁，卡片只填 `profile`、`role`、`summaryHtml`，姓名、照片、連結與入學年由 `PEOPLE_PROFILES`／`data/alumni.js` 帶入；若尚無個人頁，才在卡片資料中填 `id`、`name`、`numHtml`、`photo`。若人物誌顯示名稱與名錄正式名稱不同（例如舞台名優先），需補 `officialName` 對應名錄名稱。
- **與人物頁的長期關係（新增，2026-07-04）**：長期目標是「個人專頁」作為單一人物最完整介紹；`data/alumni.js` 的 `desc` 只放名錄卡短摘要，`PEOPLE_FEATURED_SECTIONS` 的人物誌卡片放較長策展摘要，個人頁放完整履歷、故事、來源與校正聲明。資料尚未完整時，可以先只有名錄卡；資料足夠後再建立個人頁；具代表性者再進人物誌。

### 5.1-0 人物誌首頁卡片欄位（新增，2026-07-05）
`PEOPLE_FEATURED_SECTIONS` 是人物誌首頁 `people.html` 的單一資料來源。它管理分區、卡片順序、角色標籤與首頁摘要；`people.html` 由 `scripts/generate-people-index.js` 產生，不直接手改。

| 欄位 | 使用位置 | 是否必填 | 說明 |
|---|---|---|---|
| `title` | 分區 | 是 | 人物誌首頁分區標題，如「指揮與樂團核心」。 |
| `note` | 分區 | 否 | 分區內部備註，只輸出為 HTML comment，不顯示在公開頁面。 |
| `items` | 分區 | 是 | 該分區卡片清單；陣列順序就是 `people.html` 顯示順序。 |
| `afterHtml` | 分區 | 否 | 分區後方補充文字，目前用於人物誌頁尾說明。可放 HTML，但應避免複雜結構。 |
| `profile` | 卡片 | 有個人頁者必填 | 填 `PEOPLE_PROFILES` 的 `num`，例如 `"6401"`；姓名、照片、入學年、個人頁連結會自動帶入。 |
| `role` | 卡片 | 是 | 人物誌卡片上的一句話角色標籤，需短而穩定。 |
| `summaryHtml` | 卡片 | 是 | 人物誌卡片摘要，可放少量內部連結；健康檢查會檢查可視總長與中文主體長度。 |
| `id` | 卡片 | 無個人頁者必填 | 沒有 `profile` 時作為卡片錨點，例如 `hsiao` 或 `7202`。 |
| `name` | 卡片 | 無個人頁者必填 | 沒有 `profile` 時的卡片顯示名稱。 |
| `numHtml` | 卡片 | 無個人頁者必填 | 沒有 `profile` 時的編號列 HTML，可填 `&nbsp;` 或含 `<small>` 的入學年資料。 |
| `photo` | 卡片 | 無個人頁者必填 | 沒有 `profile` 時的圖片路徑，例如 `assets/img/members/7202.webp`。 |
| `officialName` | 卡片 | 視情況 | 人物誌顯示名稱與 `data/alumni.js` 正式名稱不同時才填，例如舞台名優先顯示。 |

維護流程：

1. 有個人頁的人物：先確認 `PEOPLE_PROFILES` 與 `data/alumni.js` 已有資料，再於 `PEOPLE_FEATURED_SECTIONS` 新增 `profile`、`role`、`summaryHtml`。
2. 尚無個人頁但需要進人物誌者：新增 `id`、`name`、`numHtml`、`photo`、`role`、`summaryHtml`，必要時補 `officialName`。
3. 修改完成後一律執行 `node scripts/generate-people-index.js`，再執行 `node scripts/check-site.js`。
4. 若健康檢查出現摘要長度 warning，先依頁面視覺與資訊重要性調整摘要；不要只為了通過數字而刪除必要脈絡。

### 5.1-A 大型校友名錄呈現規劃（新增，2026-07-06）
`roster.html` 長期會承載 500 位以上校友，不應只依賴單一卡片牆瀏覽。長期定位是「可搜尋、可篩選、可切換檢視的公開索引」。

維護與設計原則：

1. **搜尋優先**：名錄頁必須能搜尋姓名、編號、入學年、聲部、角色與簡述；讀者已知道要找誰時，不應被迫捲完整頁。
2. **篩選保留**：目前「依字頭／依入學年代」與「依聲部」篩選保留，未來可加「有個人頁」「有照片」「幹部」等快速篩選。
3. **雙檢視**：卡片模式保留人物照片與摘要，適合瀏覽；列表模式用於大量資料快速掃描，適合 500+ 名錄。
4. **資料仍單一來源**：無論卡片或列表，都由 `data/alumni.js` 渲染，不建立另一份名錄資料。
5. **不新增敏感資料**：搜尋與列表只使用已公開欄位，不因功能需求加入未確認公開的內部名冊資料。

目前短期實作方向：先新增搜尋框與卡片／列表檢視切換；待公開名錄接近 500 筆時，再評估群組收合、載入更多或分頁。

### 5.1-B 在校生版面規劃（新增，2026-07-06）
在校生是嘉中管樂的現役主體，但公開網站呈現時必須注意隱私、未成年資料與每年異動。短期不建立完整在校生個人名錄，不公開班級、學號、聯絡方式或完整成員名單。

建議呈現層級：

1. **團體介紹**：`about.html` 的「在校生：嘉義高中管樂社」作為短期現役樂團入口。
2. **活動紀錄**：最新消息與影像館呈現團練、活動、演出、幹部交接與校友聯演籌備。
3. **現役樂團專區**：未來資料足夠時可新增 `students.html`，但不立即加入第一層導覽；先由首頁、關於頁或消息導流。

若公開現任幹部，應以職務介紹為主，個人資料從簡，且需確認可公開；在校生資料若要轉入校友名錄，應等畢業或取得明確公開同意後，依 `校友資料管理與驗證流程.md` 處理。

### 5.1-1 人物資料標準化欄位（新增，2026-07-04）
目前仍以 `data/alumni.js` 既有欄位維護，短期內不強迫改動資料格式；但未來若要資料化個人頁、人物誌卡片與名錄卡，應以以下欄位作為標準。

| 欄位 | 說明 | 是否公開 | 備註 |
|---|---|---|---|
| `num` | 嘉中管樂編號 | 是 | 無編號者填 `null`，不可硬補 |
| `name` | 姓名 | 是 | 若有藝名或別名，姓名欄可保留正式姓名並在別名欄補充 |
| `displayName` | 顯示名稱 | 是 | 如「吳俊霖（伍佰）」 |
| `year` | 入學民國年 | 是 | 用於排序，民國 100 後仍填完整民國年，如 104 |
| `part` | 聲部顯示文字 | 是 | 幹部需寫「職稱．聲部」 |
| `tags` | 篩選標籤 | 是 | 可複選，如 `["幹部","長號"]` |
| `roles` | 角色分類 | 是 | 如「指揮」「演奏家」「教育者」「校友行政」「社會知名校友」 |
| `role` | 名錄卡一句話頭銜 | 是 | 短句，不放過長履歷 |
| `rosterSummary` | 名錄卡摘要 | 是 | 約 40 個中文字 |
| `peopleSummary` | 人物誌卡片摘要 | 是 | 約 115-125 個中文字 |
| `profileSummary` | 個人頁頁首摘要 | 是 | 可比人物誌稍完整 |
| `fullBio` | 個人頁完整介紹 | 是 | 未來可由 Markdown 管理 |
| `photo` | 大頭照檔名 | 是 | 無照片用 `blank` |
| `link` | 個人頁或人物誌錨點 | 是 | 有個人頁優先連個人頁 |
| `featured` | 是否進人物誌 | 是 | 未來資料化 people.html 時使用 |
| `profileStatus` | 個人頁狀態 | 可公開或內部 | 建議值：`none`、`draft`、`published`、`needs-review` |

短期維護方式：

1. 新增校友時仍先加到 `data/alumni.js`。
2. 若資料只足以公開基本資訊，先不建立個人頁。
3. 若已有完整資料與來源，建立 `people/{編號}.html`。
4. 若人物具代表性或故事完整，再加入 `data/people-profiles.js` 的 `PEOPLE_FEATURED_SECTIONS` 人物誌分類，並執行 `node scripts/generate-people-index.js`。
5. 每次新增或升級人物資料，都要記錄在 `網站修改歷程紀錄.md`。

### 5.1-2 編號小遊戲資料（data/number-lookup.js）與公開名錄分流（新增，2026-07-04）
`data/number-lookup.js` 是「編號」頁查號小遊戲使用的資料，並非完整公開名錄，也不能視為全部已驗證校友資料。它與 `data/alumni.js` 分開是正確做法，原因如下：

- `data/alumni.js`：公開名錄與人物資料來源，資料應相對可信、可公開、可被搜尋引擎與讀者理解。
- `data/number-lookup.js`：查號小遊戲資料，包含尚未完全驗證的編號、姓名、聲部或幹部資訊，主要用途是互動查詢與文化展示。

管理原則：

1. 不可自動把 `data/number-lookup.js` 的人物新增到 `data/alumni.js`。
2. 若要從查號資料升級到公開名錄，必須至少確認：姓名、編號、入學年、聲部、是否同意公開。
3. 若資料只適合查號遊戲，不應建立人物頁，也不應進人物誌。
4. 查號資料若含未確認或可能不宜公開資訊，應優先評估是否拆成公開版與內部版。
5. 未來若建立自動化檢查，應檢查 `data/alumni.js` 與 `data/number-lookup.js` 的同編號資料是否衝突，但衝突不代表自動覆蓋，必須人工判斷。

建議資料狀態：

| 狀態 | 意義 | 可否進公開名錄 |
|---|---|---|
| `verified` | 已確認姓名、編號、聲部與公開狀態 | 可以 |
| `partially-verified` | 部分欄位可信，仍有缺漏 | 視缺漏內容決定 |
| `unverified` | 尚未確認 | 不建議 |
| `internal-only` | 僅供內部整理或查核 | 不可 |
| `pending-consent` | 資料可能正確，但尚未確認公開同意 | 不可 |

### 5.2 最新消息（data/news.js → 首頁最新 2 則＋校友聯演頁全列表）
- 發文流程：複製 `news/_template.html` → 改名 `news/YYYY-MM-DD-主題.html` 編輯 → 在 data/news.js 陣列**最前面**加一筆
- 資料採 `.js` 而非 `.json`：確保雙擊 HTML 離線預覽（file://）也能運作，禁止改成 fetch JSON

### 5.2-1 最新消息資料化流程（新增，2026-07-04）

8. 檢查圖片路徑、內部連結、人物連結與禁忌文案。
9. 記錄到 `網站修改歷程紀錄.md`。

建議消息資料欄位：

| 欄位 | 說明 |
|---|---|
| `date` | 發布日期，YYYY-MM-DD |
| `title` | 消息標題 |
| `summary` | 一句摘要，供列表、RSS、meta description 使用 |
| `url` | 文章路徑 |

### 5.3 長期資料化與 AI 協作方向（新增，2026-07-04）
目前已有 `data/alumni.js`、`data/news.js`、`data/number-lookup.js` 三類資料檔。這些資料檔是未來 AI 協作與自動化產生頁面的基礎，但在尚未建立產生器前，仍需維持既有 `.js` 格式以支援離線預覽。

長期資料化原則：

- **人物資料**：未來應逐步形成單一來源，避免同一人資料散落在 `data/alumni.js`、`people.html`、個人頁與聯演頁中卻不同步。
- **聯演資料**：未來應建立每屆聯演的結構化欄位，包含屆次、年份、主題、日期、場地、主辦字頭、指揮、曲目、演出人員、行政團隊、票務、海報、相簿、資料來源與資料狀態。
- **消息資料**：未來可由結構化資料自動產生文章頁、`data/news.js`、`feed.xml` 與 `sitemap.xml`，減少人工漏改。
- **相簿資料**：未來應建立相簿索引，記錄日期、活動類型、關聯屆次、場地、人物、攝影者、精選照片與外部完整相簿。
- **查號資料**：`data/number-lookup.js` 資料量大於公開名錄，未來應明確區分「公開查詢資料」與「內部維護資料」，不得把未確認同意公開的個資放上公開網站。
- **AI 協作**：AI 可以協助整理草稿、補欄位、產生摘要與檢查一致性，但不得跳過資料來源、本人同意、隱私界線與修改歷程紀錄。

未來若導入資料產生器，輸出結果仍應是可部署到 GitHub Pages 的靜態 HTML／CSS／JS，不應在沒有必要時導入後端服務。

## 六、內容準則（史實與文化正確性）

以下為已查證的設定，**所有新文案必須遵守**：

1. **編號取得時機**：高一入隊時沒有編號，高一升高二幹部交接時才獲得。禁止寫「進隊第一天就有編號」
2. **編號排序**：凡並列呈現多位成員，一律依入學年由前輩至後輩（0 開頭視為民國百年後、排最後；無編號但世代明確者依世代排）
2-2. **people.html 卡片排序**：頁首說明「各區依入學年由前輩至後輩排列」——每個 `<div class="cards">` 分類區塊內，卡片一律依該人**實際民國入學年**（`data-alumni.js` 的 `year` 欄位，非編號表面數字）由小到大排列；同一年份再依四位數編號末兩碼由小到大排列。切勿直接比較編號字串大小──民國百年後入學者編號常以 `0` 或 `1` 開頭（如 0431＝104 年、1051＝110 年），字面數值遠小於「63」「85」等兩位數編號，但入學年份其實更晚，必須換算實際年份後才能排序，不可依編號表面值排序。新增或搬動卡片後，務必重新檢查該區塊順序。
2-1. **字頭定義（重要，曾勘誤）**：字頭＝**編號第二碼**（入學民國年的個位數）。7581、8501、9502、05XX 皆為「五字頭」，同字頭每十年一輪、跨世代同門。**禁止**寫成「編號第一碼」或「民國 X0 年代」。近年聯演由字頭依序輪值主辦（2025 四字頭《四方之音》、2026 五字頭《為伍》），主辦數字常藏於主題名稱；輪值慣例起點待考，更早年代為熱心學長姐主動承辦
3. **音樂廳**：2026《為伍》並非首度在文化局音樂廳演出。海報可考屆別：28（2012）、31（2015）、34（2018）、36（2020）。寫「睽違六年重返」，禁寫「首度移師」
4. **售票**：並非首度售票，2015、2018、2020 均售票（100 元、兩廳院系統／ibon）。禁寫「首度公開售票」，可寫「透過 OPENTIX 售票」
5. **屆數對照**：2012＝28、2014＝30、2015＝31、2017＝33（推算）、2018＝34（推算）、2020＝36、2021 疫情停辦、2022＝37、2023＝38、2024＝39、2025＝40、2026＝41。推算值若日後獲得節目冊佐證有誤，需同步改 concerts.html 與本表
6. **稱謂**：在校生社團稱「嘉義高中管樂社」，校友組織稱「嘉義高中校友管樂團」（2008 立案），合稱「嘉義高中管樂隊暨校友管樂團」
7. **暫定資訊**：曲目、票價等籌備中內容必須標註「（籌備中，實際以正式公告為準）」
8. **個資界線**：網站僅揭露編號、姓名、入學年、聲部與公開經歷；Email、電話、地址等聯絡資料一律不上網

### 7.1 每頁固定結構

- 網站圖示：favicon.ico（根目錄）＋ assets/img/favicon-32.png／icon-192.png／apple-touch-icon.png／icon-512.png，源自社徽（深藍底嘉中瓶徽）

### 7.2 人物個人專頁（人物誌的第三層）
- 架構：人物誌卡片牆（people.html）→ 個人專頁 `people/{編號}.html`（如 people/7581.html），名錄卡 `link` 也可直接指向個人頁；導覽 active 固定「人物誌」，資源路徑加 `../`
- 人物誌首頁維護：`people.html` 由 `scripts/generate-people-index.js` 依 `data/people-profiles.js` 的 `PEOPLE_FEATURED_SECTIONS` 產生；卡片摘要、角色、分區與無個人頁卡片資料不可直接在 `people.html` 手改。
- 內容結構：page-head（編號＋姓名）→ 人物摘要表（編號、聲部、入學年、現職）→ 分節故事 → 資料來源列表 → 校正聲明（「本文整理自公開資料與本人自述，如需更正請聯繫我們」）
- 撰寫原則：**「目前可確認的身分」與「特定年份的紀錄」分開表述**，歷史資料標明年份（如「2013 年公開資料記載…」），不把舊資料寫成現況；未成年或未確認同意者的家屬姓名不刊出；刊出前務必經本人過目
- 新增個人頁記得加進 sitemap.xml
- **全站姓名自動連結（新增，2026-07-04）**：只要某人已有個人專頁（`people/{編號}.html`），**全站任何頁面的敘述文字（`<p>`、`<li>`、`<blockquote>`、`<figcaption>` 等內文段落）中每次提到其姓名，都必須加上連結指向該人個人專頁**（相對路徑記得依頁面深度加 `../`）。適用範圍包含 concerts.html 與歷屆聯演獨立頁的簡介段落、history.html、numbers.html、about.html、影像館／最新消息的圖說與內文，以及人物誌（people.html）卡片、其他人個人專頁內文提到的第三人。
  - 例外（不連結）：`<title>`、`<meta>`、`<script>`、HTML 註解、`alt`／`href`／`src` 等屬性值——這些不是可視敘述文字
  - 例外（不連結）：某人**自己的**個人專頁內文提到自己姓名時不自我連結（已在當頁，連結無意義）；卡片／段落標題（如 `<h3>姓名</h3>`）本身是標籤而非敘述文字，也不需連結
  - 若文字中已經是連結（如歷屆聯演名單表格 `<td>` 內的姓名），不重複包一層 `<a>`
  - 尚未有個人專頁者（僅在 roster.html／人物誌有卡片），姓名維持純文字，等該人建立個人專頁後才開始套用此規則
  - 新增個人專頁或新校友資料時，記得回頭檢查全站是否有該姓名的既有提及需要補上連結

### 7.3 年度更新流程（每年聯演循環）
1. 籌備期：發最新消息（售票、團練探班等）、更新當屆區塊與購票連結
2. 演出後：當屆從 concerts.html「本屆演出」改寫為「歷屆紀錄」項（含海報縮圖）；影像館開新相簿；時間軸視情況加節點
3. 幹部交接：發交接消息、名錄補新任幹部
4. 資料豐富的屆別可另開獨立頁（建議路徑 `concerts/第N屆.html`，套用文章頁版型）

## 八、技術限制與部署

- 純靜態網站（HTML／CSS／原生 JS），無建置工具、無框架、無外部 JS 函式庫；唯一外部資源是 Google Fonts
- 必須支援離線雙擊預覽（file://），因此禁用 fetch/XHR 載入本地資料
- 部署：GitHub Pages（步驟見 README）；全站容量目標 <100MB（照片按第四章規格壓縮即可安全達標）
- 相容性基準：近三年之 Chrome／Safari／Edge／行動瀏覽器

### 8.1 自動化檢查方向（新增，2026-07-04）
目前尚未建立正式檢查腳本，但未來若網站持續擴張，應優先建立可由人工或 AI 修改後執行的檢查流程。建議檢查項目：

- HTML 內部連結與圖片路徑是否存在。
- `data/alumni.js`、`data/news.js`、`data/number-lookup.js` 是否語法正確。
- 新增頁面是否同步加入 `sitemap.xml`。
- 新增消息是否同步加入 `data/news.js` 與 `feed.xml`。
- 已有個人專頁者，全站敘述文字中的姓名是否連結。
- 是否出現第六章列出的禁忌文案或史實錯誤。
- 圖片是否為 WebP、尺寸與命名是否符合第四章規範。
- 是否有 `.DS_Store` 等不應上傳的系統檔案。
- 是否已在 `網站修改歷程紀錄.md` 記錄本次修改。

自動化檢查不取代人工史實判斷；涉及人物資料、待考歷史、演出曲目、個資與本人同意時，仍需人工確認。

## 九、修改歷程紀錄規範（新增，2026-07-04）

`網站修改歷程紀錄.md` 是本站長期維護的正式紀錄檔。自 2026-07-04 起，未來每一次修改網站，原則上都必須在該文件新增一筆紀錄，作為之後追蹤問題、回顧架構階段、與 AI 協作時理解上下文的依據。

### 9.1 必須記錄的修改類型

以下修改一定要記錄：

- 新增、刪除或改寫任何公開頁面。
- 新增、刪除或改寫人物資料、校友名錄、編號查詢資料。
- 新增、刪除或改寫最新消息、RSS、sitemap、SEO meta。
- 新增、刪除或改寫歷屆聯演、相簿、圖片資料。
- 修改 CSS、JavaScript、版面、響應式規則或互動功能。
- 修改本規範、README、知識整理文件、優化報告或維護文件。
- 導入任何自動化腳本、模板化流程、資料化流程或 AI 協作流程。
- 修正重大史實、稱謂、屆數、售票、音樂廳、個資或授權問題。

### 9.2 建議記錄欄位

每筆紀錄建議至少包含：

- 日期時間。
- 修改階段或版本。
- 修改者。
- 修改原因。
- 修改檔案。
- 修改內容摘要。
- 影響範圍。
- 驗證方式。
- 後續待辦。
- 備註。

### 9.3 修改工作完成定義

未來任何修改若涉及公開網站或維護規範，必須同時滿足以下條件，才視為完成：

1. 檔案已修改。
2. 必要檢查已完成。
3. 若影響公開頁面，已檢查手機與桌面顯示。
4. 若新增頁面或消息，已同步 SEO／sitemap／feed 等必要檔案。
5. 已在 `網站修改歷程紀錄.md` 新增紀錄。

若因為緊急狀況先修改網站，仍應在事後補登修改紀錄，並註明為補登。
