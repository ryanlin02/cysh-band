## 二、人物頁定位

人物頁是單一人物最完整的公開介紹頁，職責高於名錄卡片與人物誌卡片。

| 層級 | 檔案 | 職責 |
|---|---|---|
| 校友名錄卡 | `data/alumni.js` → `roster.html` | 短摘要、搜尋與索引 |
| 人物誌卡片 | `people.html` | 精選人物的策展入口 |
| 人物個人頁 | `people/{編號}.html` | 最完整介紹、來源、補充連結、聯演紀錄 |

原則：

1. 人物個人頁是「完整資料與故事」的主要承載處。
2. `data/alumni.js` 的 `desc` 不應塞長履歷，只放名錄卡短摘要。
3. `people.html` 卡片只放人物誌策展摘要，不取代個人頁。
4. 個人頁可以存在但不一定要進 `people.html` 人物誌，例如資料完整但尚未達策展門檻者。

## 三、標準頁面區塊

未來模板化後，每個人物頁建議固定由以下區塊組成。

### 3.1 Head / SEO

必要欄位：

- `title`
- `description`
- `og:type`
- `og:title`
- `og:description`
- `og:url`
- `og:image`
- `og:image:width`
- `og:image:height`
- `twitter:card`

建議：

- `og:type` 短期可維持 `profile`。
- `og:image` 短期維持全站共用 `https://cysh.band/assets/img/og.jpg`。
- 若未來有個人專用社群分享圖，再改為個人圖。

### 3.2 頁首人物資料

固定內容：

- `PEOPLE．{編號}`
- H1：人物姓名＋一句主題式標題
- 大頭照
- 基本資料表

基本資料表建議欄位：

- 編號：含民國入學年與聲部
- 主修／主要樂器
- 學歷（如適用）
- 現職（如適用）
- 樂團角色（如適用）
- 綽號（如適用）

欄位不必每頁完全相同，但要從同一組候選欄位中挑選，避免每頁任意命名。

### 3.3 正文介紹

正文建議架構：

1. 開場摘要：用一段文字說明此人物與嘉中管樂的核心關係。
2. 成長或入隊背景。
3. 專業發展或主要經歷。
4. 與嘉中／校友團的連結。
5. 現況或長期影響。
6. 如有代表性語句，可放 `blockquote`。

小標題以 `<h3>` 呈現；不建議跳到 `<h2>`，因為文章外層已由頁面結構承接。

### 3.4 相關校友聯演

建議新增固定區塊：

```html
<h3>相關校友聯演</h3>
```

短期資料來源可由 `data/concerts.js` 產生，但必須標示：

> 以下為目前資料庫可確認的相關聯演紀錄，並非此人物完整參與履歷。

原因：

- `data/concerts.js` 目前仍是資料化試作版。
- 許多早期聯演名單尚未完整建檔。
- 部分人物可能長期參與但尚未被結構化資料記錄。

相關聯演可包含：

- 擔任指揮
- 擔任獨奏／協奏
- 擔任召集、總召、製作、行政統籌
- 作品首演／委託創作
- 明確出現在該屆頁面或資料檔的角色

### 3.5 相關連結

建議新增固定區塊：

```html
<h3>相關連結</h3>
```

可收錄：

- 官方人物頁
- 樂團成員介紹
- 學校／機構師資頁
- 演出節目頁
- YouTube 公開演出影像
- 新聞報導
- 作品或專輯頁

不建議收錄：

- 私人電話、住址、email
- 未經本人同意的私人社群帳號
- 來源不明或疑似搬運的內容
- 可能造成個資風險的內部名冊截圖

### 3.6 資料來源

資料來源應與相關連結分開：

- 「相關連結」偏向讀者延伸閱讀。
- 「資料來源」偏向本文依據與查核脈絡。

建議資料來源分級：

| 類型 | 說明 | 呈現方式 |
|---|---|---|
| 公開來源 | 官方網站、新聞、節目頁、公開影片、論文系統 | 可連結 |
| 本人提供 | 本人填表、自述、校友訪談 | 可寫文字註記，不公開內部表單 |
| 校友團紀錄 | 節目冊、演出名單、社群保存資料 | 可寫紀錄類型，視授權決定是否連結 |
| 待確認 | 尚未取得完整佐證 | 明確寫「尚待確認」 |

資料來源文字應避免讓讀者誤以為所有敘述都已百分之百驗證。可使用：

> 本文整理自以下公開資料與校友團保存紀錄；部分早期經歷尚待本人或原始節目冊進一步確認。

## 四、圖片規則

### 4.1 大頭照

沿用既有規格：

- 目錄：`assets/img/members/`
- 格式：WebP
- 尺寸：正方形 480×480
- 檔名：以編號命名，例如 `8861.webp`

無照片時：

1. 短期：沿用 `blank.webp`。
2. 中期：依聲部使用樂器特寫預設圖，例如法國號、小號、低音號。
3. 長期：本人授權後補上大頭照。

### 4.2 內文圖片

未來可支援內文圖片，但不強制每頁都有。

適合圖片：

- 演出照片
- 節目冊或海報局部（需確認可公開）
- 樂器／排練現場
- 本人授權照片

格式建議：

```html
<figure>
  <img src="../assets/img/people/{編號}/{檔名}.webp" alt="具體描述">
  <figcaption>照片說明。</figcaption>
</figure>
```

內文圖片不得取代資料來源；圖片本身若涉及授權，應在資料來源或圖說中說明。

## 五、資料模型建議

短期不急著把 19 頁全部改成 JSON 或 Markdown；但模板化時可先採「內容片段＋資料清單」方式。

建議未來結構：

| 檔案 | 用途 |
|---|---|
| `content/people/{編號}.html` | 人物頁正文片段；目前 19 個人物頁皆已建立來源檔 |
| `data/people-profiles.js` | 人物頁標題、SEO、基本資料、相關連結、來源等結構化資料；目前已建立 |
| `scripts/generate-people-pages.js` | 由資料與正文產生 `people/{編號}.html`；目前產生全部 19 個人物頁 |
| `people/{編號}.html` | GitHub Pages 實際部署的正式 HTML |

短期也可先不新增 `data/people-profiles.js`，改在 `scripts/generate-people-pages.js` 內放少量試點資料；待格式穩定後再拆出資料檔。

目前已從短期方案推進到資料檔方案：頁面正文放在 `content/people/`，人物 metadata 放在 `data/people-profiles.js`，正式 HTML 由 `scripts/generate-people-pages.js` 產生。

在正式模板化前，可先把下列項目逐步加入 `scripts/check-site.js`。

### 6.1 目前可立即檢查

- [x] 每個 `people/*.html` 都有 `person-hero`。

- [x] 每個 `people/*.html` 都有 `資料來源`。

- [x] 每個人物頁都能從 `data/alumni.js` 找到對應 `link`。
- [x] `data/alumni.js` 的人物頁 `link` 若指向 `people/{編號}.html`，檔案必須存在。
- [x] 頁首圖片檔案必須存在，且要有非空 `alt`。
- [x] `og:type` 應為 `profile`。
- [x] `og:image:width` / `og:image:height` 應補齊。

上述已勾選項目已於 2026-07-04 至 2026-07-05 陸續加入 `scripts/check-site.js` 與人物頁模板流程。`og:image:width` / `og:image:height` 已隨人物頁全數模板化補齊。

### 6.2 模板化後可檢查

- `content/people/{編號}.html` 與正式 `people/{編號}.html` 是否同步。
- `data/people-profiles.js` 是否包含每個模板化人物頁的必要欄位。
- 每頁是否正好有一個「相關校友聯演」區塊。
- 每頁是否正好有一個「相關連結」或明確標示無相關連結。
- 每頁資料來源是否標示來源類型。
- 若使用 `blank.webp`，是否有對應聲部預設圖待補註記。

### 6.3 不建議自動判斷的項目

- 人物成就是否足以進人物誌。
- 某筆聯演是否為該人物完整參與紀錄。
- 來源是否足以支持每一句履歷敘述。
- 是否適合公開私人社群或聯絡方式。

這些仍需人工判斷，健康檢查只能提示缺漏，不應自動修改。
